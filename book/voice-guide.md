# Voice Guide — 全書寫作風格守則

> 所有章節必須遵循。違反者 QA 退回。

## 1. 不是教科書,是戰場筆記

- 每章開頭以「冷觀察」起手:一個真實/擬真現場片段、一句被嗆的話、一個帳單、一張圖。
- **禁用開場句**:「本章將介紹…」「在本章中,我們將學習…」「隨著科技進步…」
- **建議開場句**:「我在 {年份} 看過一個 {角色} {做了什麼}…」「{某個會議上} 有人問了一句…」

## 2. 四段式節奏(硬規)

每章必須包含、且依此順序:

1. **冷觀察** — 現場、衝突、引子
2. **真問題** — 把表面需求拆開來看,指出真正在處理的是什麼
3. **決策框架 / 模式** — 可抄走用的判準、表格、Mermaid
4. **踩坑清單 + 交付清單** — 常見地雷與可帶走的 artifact 模板

## 2.1 Tonal Arc(語氣弧線,硬規)

戰場筆記的張力**來自對比**。整章都緊繃會讓讀者疲勞,所以四段必須對應四種語氣:

```
冷觀察          真問題          決策框架        踩坑 + 交付
─────────       ─────────       ─────────       ─────────
TENSE           STEADY          COACHING        CONSTRUCTIVE
戰場            穩定            教練            建設

「被嗆的話」    「讓我把它       「這幾個維度    「拿這個模板,
「沒人答得      拆開來看」       可以幫你         一頁就夠」
 出來」         「換句話說」     判斷」          「先做這一件,
                「真正在處理     「我在現場       再讀下一章」
                 的是什麼」      常用的是…」
```

**核心原則**:張力只在開頭一次。中段是「老師傅在白板前慢慢畫給你看」的節奏;尾段是「給你工具、相信你做得到」。

### 各段語氣的詞彙約束

| 段 | 語氣標籤 | 允許用語 | 禁止用語 |
|---|---|---|---|
| §1 冷觀察 | **TENSE** | 「我看過」「他沒答上來」「帳單寄到」「沒人能解釋」 | 「業界普遍」「一般而言」「眾所周知」 |
| §2 真問題 | **STEADY** | 「讓我把它拆開」「換句話說」「真正在處理的是」「這件事其實是…」 | 「我不接受」「這是錯的」「拆穿」「徹底失效」 |
| §3 決策框架 | **COACHING** | 「我在現場常用」「這個維度是」「你可以試」「我給的最小劑量」 | 命令式、批判式、「絕對」「必須這樣」「不能那樣」 |
| §4 踩坑 | **CONSTRUCTIVE** | 「常見地雷」「修正方向」「這樣做會穩」「下次遇到可以這樣處理」 | 「白癡」「廢」「沒救」式發洩、嘲諷 |
| §4 交付 | **CONSTRUCTIVE** | 「拿這個模板」「一頁就夠」「先做這一件」「你應該已經能…」 | 「最佳實踐」「銀彈」「萬用解法」 |

### 弧線檢查清單(QA 必查)

- [ ] 冷觀察結束處讀起來有「想往下看」的拉力(TENSE 達標)
- [ ] 真問題段沒有怒氣或拆穿語氣(STEADY 達標)
- [ ] 決策框架段是「我教你怎麼判斷」而非「我告訴你答案」(COACHING 達標)
- [ ] 踩坑段每個反模式都配「修正方向」blockquote(CONSTRUCTIVE 達標)
- [ ] 交付段以「相信讀者做得到」結尾,不是以警告結尾(CONSTRUCTIVE 達標)

### 例外

- **補章類**(已寫好的 A–F)允許整體語氣略硬,因為定位為「戰場筆記」加強版。
- **Ch 1 / Ch 39 / Capstone** 等定錨章節可在交付段提高情緒強度(因為是篇尾的勵志收束)。

## 3. 第一人稱使用規則

第一人稱用得太密,讀起來會像被教訓而不是被指引。**「我」只在它能換取讀者信任的位置出現**,其他位置一律改成中性觀察或第二人稱建議。

### 「我」的使用分級

| 等級 | 使用位置 | 例 | 頻次上限 |
|---|---|---|---|
| **A 必要** | 真實見證敘事(冷觀察段) | 「我在 2025 年看過一個案例」「他沒答上來」 | 每章 ≤ 3 次,集中在 §1 |
| **B 可選** | 引用他人說的話、敘事中的我 | 「CTO 問了一句話,我把它原樣記下來」 | 每章 ≤ 2 次 |
| **C 禁用** | 表達意見、給建議、下判斷 | ❌「我給自己的拇指法則是」「我堅持」「我不接受」「我建議」 | 0 次 |

### 替換對照

| ❌ 教訓口氣(C 級) | ✅ 指引口氣 |
|---|---|
| 我給自己的拇指法則是 | 一個好用的拇指法則是 / 拇指法則可以這樣定 |
| 我在現場常用的是 | 現場常用的是 / 比較穩的做法是 |
| 我觀察到 | 常會看到 / 在現場容易看到 |
| 我建議 | 建議 / 你可以試 / 從這一件先做 |
| 我堅持 / 我不接受 | (整句改寫,不用個人立場句) |
| 我看過這樣會出事 | 這樣做容易出事 / 現場很常出事 |
| 讓我把它拆開來看 | 把它拆開來看 / 拆開來看 |

### 「我們」的使用

- ❌ 禁止「我們應該 / 我們不能」這種替讀者做決定的句式
- ✅ 允許出現在被引用的對話、被引用的口號、模板欄位(如 System Charter 的「我們在解決什麼」這類**讀者填寫位**)

## 4. 每章硬規

- ≥ 1 Mermaid 圖
- ≥ 1 決策表
- ≥ 1 案例(已註冊 `_refs/case-registry.yaml`)
- ≥ 1 反模式(Anti-Pattern)
- ≥ 1 可帶走 artifact 模板(章末交付清單)

## 5. 案例輪替

對應 `.claude/skills/shared/domain-*`:
- **energy** — 儲能、光電、EMS、電網調度
- **fintech** — 支付、清算、風控、KYC
- **healthcare** — EMR、HL7、FHIR、病歷
- **ecommerce** — 訂單、庫存、物流、行銷
- **saas** — 多租戶、計費、權限

每章主案例選一個領域,反例可跨領域。**純虛構**:公司名、產品名、數字皆不實,但商業邏輯與技術細節必須真實可驗證。

## 6. 技術名詞

- 中英並陳:「事件溯源(Event Sourcing)」、「最終一致性(Eventual Consistency)」
- 第一次出現英文全寫,後續可縮:「Service Level Objective(SLO)」→ 後續用 SLO
- 版本號必標:React 19、PostgreSQL 17、OpenTelemetry 1.x、Kubernetes 1.31
- 工具/廠商首次出現括註:「Backstage(Spotify)」「ArgoCD(Intuit / CNCF)」

## 7. 禁止清單

- 大段定義式段落(連續 3 段以上純定義)
- 「五大原則」「七大要素」湊數式列表
- 沒有上下文的 bullet 海(超過 7 個 bullet 沒有分組)
- 安全廢話(「這很重要」「需要注意」「請謹慎」)
- 預測式承諾(「未來 5 年內…」「下一波趨勢將是…」)

## 8. 格式硬規

- 字數:一般章 4,000–8,000 字;補章類 6,000–10,000 字
- 章首 frontmatter 必填:見 `conventions.md`
- 章末必有 cross-ref 區塊:**前置閱讀** / **下游章節** / **延伸補章**
- Mermaid 顏色:`hot fill:#fee`(危險/反模式)、`cold fill:#eef`(現況)、`goal fill:#efe`(目標)

## 9. 引用與術語

- 新術語先進 `annex-f-glossary.md` 才能在內文使用連結
- 外部引用先進 `annex-g-citations.md` 編號(`CIT-NNN`),內文 `[^CIT-NNN]`
- 案例先進 `_refs/case-registry.yaml`(`CASE-XXX-NNN`),內文以 ID 引用

## 10. 寫作節奏建議

- 段落短,3–5 句一段為主。連續長段必有原因(複雜推理、敘事高潮)。
- 對話/引言獨立成段,用 `>` blockquote。
- 表格 > bullet list > 連續 prose,優先順序如上。
- 程式碼片段必標語言、必標版本、必可獨立運行(或註明依賴)。
