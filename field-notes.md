# 第 26 章｜邊緣計算與 OT/IT 融合的系統架構

> *Edge Computing & OT/IT Convergence — 給有實體設備的系統*

## 26.1 一個雲原生工程師遇到 200kW/400kWh 儲能櫃時會怎麼錯

我在現場看過一個典型情境。一位雲端架構師被指派去設計一個結合儲能(ESS)、太陽能逆變器、與電網的能源管理系統。他畫了漂亮的微服務拓樸,Kafka 串流、PostgreSQL 主從、Kubernetes 叢集、Grafana 儀表板,一應俱全。

到了 PoC 現場,設備櫃旁的工程師問了一句:**「網路斷了三十秒,你的系統會把電池放掉嗎?」**

他沒答上來。

這個問題揭穿了一個根本盲點:當你的系統包含**會傷人、會燒掉、會違反電力法規**的硬體時,你不是在設計軟體系統,你是在設計一個「軟體只是其中一層」的更大系統。儲能櫃的 BMS(電池管理系統)不會等你的 Kubernetes Pod 重新調度。電網調度單位的 EMS 不會接受「我們的 Kafka 集群正在重啟,請等三分鐘」這種理由。

主大綱第 23–24 章談雲原生與 Kubernetes,前提是「網路是可靠的、計算資源是隨叫隨到的」。這個前提在 OT 世界**完全不成立**。

## 26.2 OT 與 IT 的邊界:不要試圖把它消除

新手最常犯的錯,是把「OT/IT 融合」誤解成「把 OT 雲端化」。正確的理解應該是:**讓兩個世界在一個受控的邊界上對話,而不是把較弱的一方拖進另一方的假設裡**。

| 面向 | OT 世界 | IT 世界 |
|---|---|---|
| **時間尺度** | 毫秒(控制迴路)~ 秒(SCADA 巡檢) | 秒 ~ 分鐘 |
| **故障容忍** | 不能停;停了會傷人或違法 | 可降級、可重試、有 SLA |
| **生命週期** | 設備 10–20 年,韌體可能五年沒動 | 服務一年迭代多次,容器分鐘級更新 |
| **網路假設** | 工業以太網、Modbus、CAN、有時是 4G/5G,常斷線 | TCP/IP 永遠通、頻寬大致穩定 |
| **安全模型** | Air-gapped 或 IEC 62443 區段隔離 | Zero Trust + mTLS |
| **失敗代價** | 設備損壞、人身傷害、罰款 | 訂單流失、客戶抱怨 |
| **首要 KPI** | Availability(可用度,5 個 9 起跳) | Latency p99 / Throughput |

這張表的關鍵是**最後一列**。OT 工程師談 KPI 的時候,單位是「年」;IT 工程師談 KPI 的時候,單位是「分鐘」。這兩個世界共用同一份系統規格,卻使用完全不同的詞彙談「可用」。

**邊界的設計原則**(這是我寫進每一份合約的那條):

1. **OT 永遠是 Source of Truth**。設備上的值是真實世界的反映,雲端是這個值的「最終一致性副本」。如果雲端與設備的值衝突,以設備為準。
2. **控制路徑必須能被切斷**。雲端可以發送建議性的調度,但 OT 端必須具備拒絕、覆寫、回退本地策略的能力。所有從 IT 往 OT 寫入的命令,必須通過一個本地閘道做安規檢查(例如 SOC、DOD、溫度上下限)。
3. **資料流(IT 讀)與控制流(IT 寫)走不同管道**。讀通常用 MQTT 或 Modbus-over-TCP,寫必須用受嚴格認證的另一條通道(常常是另一個物理 NIC + 另一套憑證)。

## 26.3 邊緣節點的離線自治(Offline Autonomy)

雲端工程師常常忽略一件事:**邊緣節點不是一個小一點的雲端**,它是一個**自己就能完成業務閉環**的微縮系統。網路只是錦上添花。

我的判準很簡單:**如果中央被 DDoS 攻擊兩天,這個邊緣節點還能照常工作嗎?**

如果答案是「不行」,那這不是邊緣計算,只是「把計算放遠一點的雲端」。

### 離線自治的四個層次

```
Level 0:  純資料採集器(Pure Data Collector)
          ↑ 中央離線就停擺
Level 1:  本地暫存與重傳(Store & Forward)
          ↑ 能撐一段時間,但業務邏輯仍在中央
Level 2:  本地策略執行(Local Policy Execution)  ← 多數能源 / 製造系統的最低標準
          ↑ 能執行預載策略,中央只負責更新策略
Level 3:  完整本地閉環(Full Local Closed-Loop) ← 高安全性場域(醫療、軍工、危險設備)的標準
          ↑ 中央只是用來分析、優化、報表
Level 4:  邊緣協作網(Edge Mesh / Peer Coordination)
          ↑ 連中央都消失,邊緣節點之間還能彼此協調
```

> 📌 [圖示建議 26-2]
> *畫一張「離線自治成熟度模型」金字塔,從 Level 0 到 Level 4 由下往上,每一層註明:典型應用、所需本地資源、典型部署成本。*

絕大多數案場我們會把目標設在 **Level 2**:預載一份「在合理範圍內可自行運轉」的策略,中央離線時邊緣繼續按舊策略走,並把所有事件記錄起來等中央回來補同步。

### 工程上要做哪些事才能達到 Level 2

- **本地時序資料庫**(TimescaleDB / InfluxDB / SQLite + 自寫 LSM)— 至少七天的本地保留。
- **本地規則引擎**(Drools 太重,通常自寫一個小的;或用 Node-RED / Rule Engine in Go)— 把調度策略以「規則 + 表」的形式下發。
- **本地 Web UI**(可選但建議)— 讓現場工程師能在沒有網路時直接到設備櫃旁邊操作。
- **狀態同步協議**(這是難點)— Append-only log + monotonic clock + idempotent replay,讓網路恢復時能無痛重放。

## 26.4 數位孿生(Digital Twin):它不是 3D 模型,是「資料契約」

數位孿生這個詞被行銷化到失去意義。把它當成「設備的 3D 渲染」是嚴重誤解。在嚴肅的工程語境裡,數位孿生是一份**資料合約**,描述:

1. 這台設備**現在的狀態**(實時遙測 — Telemetry)
2. 這台設備**應該的狀態**(設定點 — Setpoint / Desired State)
3. 這台設備**經歷過的歷程**(事件序列 — Event Log)
4. 這台設備**怎麼回應命令**(行為模型 — Behavioral Model)

這四項對應到 SA/SD 的語彙是:**讀模型、寫模型、事件溯源、領域模型**。完全不是新東西,只是被搬到一個有溫度、有電壓、有實體後果的領域。

### 數位孿生資料建模的三個錯誤

**錯誤 1:把所有資料都同等對待。** 設備上有上百個資料點(電壓、電流、溫度、SOC、SOH、告警代碼...),但只有 5–15 個是「業務上有意義」的。把這 15 個變成「業務級資料點」走高品質管線(高頻率、強保證);其餘走「審計級資料點」(低頻率、Best-effort)。混在一起會把高品質管線塞爆。

**錯誤 2:用一張寬表存所有設備狀態。** 設備之間的差異很大(逆變器 vs 電池 vs 電表),用一張寬表會欄位炸裂。用**通用標頭 + 設備類型專屬負載**的兩段式 schema,前者拿來索引、後者放原始細節。

**錯誤 3:把孿生資料庫當成業務資料庫用。** 數位孿生反映「實體世界當下狀態」,業務資料庫反映「商業實體狀態」。它們生命週期、查詢模式、合規要求都不同。要分開設計、分開儲存。

## 26.5 MQTT 的可靠性設計:不要相信 QoS 1 / QoS 2 的字面承諾

MQTT 是 OT/IT 邊界事實上的標準協定。但網路上絕大多數 MQTT 教學都停在「QoS 0 / 1 / 2 三個等級的差別」,那是入門。**真正的可靠性要從以下幾個層面構築:**

### 26.5.1 Broker 的選擇是 25% 的問題

| Broker | 適用場景 | 我的踩坑紀錄 |
|---|---|---|
| **Mosquitto** | 小規模、教學、單一 Broker | 高負載下 retained message 會掉,別用在工業場景 |
| **HiveMQ** | 中大型、商業支援、Kubernetes 友善 | 商業版穩定,社群版功能受限 |
| **EMQX** | 大規模、多協議橋接、台灣 / 中國方案 | 規模化好,但要熟悉它的叢集模型 |
| **NATS JetStream** | 不是嚴格 MQTT 但語意相近,適合混搭 IT 端 | 觀測性與重放能力比 MQTT 強 |
| **AWS IoT Core / Azure IoT Hub** | 雲端原生、設備管理、影子設備 | 鎖定雲廠 + 帳單會嚇人;設備離線時影子設備邏輯複雜 |

### 26.5.2 真正讓 MQTT 「可靠」的不是 QoS,是 Sparkplug B

Sparkplug B(Eclipse 基金會規範)是把 MQTT 從「訊息協定」升級為「狀態協定」的關鍵。它強制:

- **每個邊緣節點(EoN)在連線時必須宣告完整狀態(NBIRTH)**;之後只發增量(NDATA)。
- **節點意外斷線時,Broker 透過 LWT(Last Will and Testament)發出 NDEATH**;訂閱端立刻知道「這個節點死了」,而不是「這個節點剛好沒新資料」。
- **訊息序號(seq)強制 0–255 循環**;接收端可立即偵測丟失,觸發重新請求。

沒有 Sparkplug B,你會花掉大量時間自己重造這套機制,而且通常重造得不對。

### 26.5.3 不要把雲端業務邏輯掛在「MQTT 訊息順序保證」上

MQTT 不保證跨主題的順序、不保證跨 Broker 節點的順序、甚至同一 client 重連後的順序也不保證。**設計時假設 MQTT 訊息是 at-least-once + out-of-order**,把順序語意做在應用層(時間戳 + idempotency key + 訊息序號)。

# 第 45 章｜Agentic QA ⸺ 非確定性系統的品質保證

> *Quality Assurance for Non-Deterministic Systems*
>

## 45.1 「LGTM」不能再是一個動詞

過去的軟體 QA 有一個隱含假設:**輸入相同 → 輸出相同**。所以我們才能寫單元測試、整合測試、E2E 測試,並期待它們在 CI 中綠燈通過。

LLM 系統打破了這個假設。同一個 prompt,溫度設成 0.7,跑 100 次會得到 100 個略有差異的輸出。把 prompt 加長一個 emoji,結果可能完全變掉。把使用者的對話歷史延長到第 15 輪,Agent 開始忽略系統提示。

如果你的工程文化是「Code Review 看一眼說 LGTM、CI 跑綠就 merge」,這套對 Agent 系統根本不夠用。**Agent 系統需要的不是測試,是治理**。

對於一個用 6–7 人精銳編制取代傳統大型 R&D 部門的團隊來說(這是當前 AI 賦能組織的典型結構),Agentic QA 不是錦上添花,**是確保系統不在凌晨三點崩潰把整個團隊叫起床的最後一道防線**。

## 45.2 把 QA 重新分類:確定性層 vs 非確定性層

混亂的根源是把所有測試都丟在一起。乾淨的解法是把 Agent 系統切成兩層:

### 45.2.1 確定性層(Deterministic Layer)

包括:工具(Tool)、Function Calling 的契約、檢索器(Retriever)、外部 API 包裝、資料庫查詢、結構化輸出的 schema 驗證。

**這一層用傳統 QA 測。** 寫單元測試,寫契約測試,寫整合測試。Pact、Spectral、jsonschema、OpenAPI validation,該怎麼用就怎麼用。這部分如果你跳過,後面所有問題都會被歸咎於「LLM 不準」,但實際上是你的工具回傳格式不穩。

### 45.2.2 非確定性層(Non-Deterministic Layer)

包括:LLM 的回答品質、Agent 的決策路徑、多步推理的完整性、工具選擇的合理性、最終輸出的有用性。

**這一層用 Eval 測,不是 Test 測。** Eval 是統計性的,接受「100 次中 95 次正確就通過」的概念,而不是「綠燈紅燈」。

| 面向 | 傳統 Test | Agent Eval |
|---|---|---|
| 結果類型 | Pass / Fail | 0–100 分,或多維度向量 |
| 通過標準 | 100% 通過才能 merge | 例如「>= 90 分,且無單一維度 < 70 分」 |
| 失敗代價 | 單個案例就阻擋 | 整體分數下降才阻擋(Eval Regression) |
| 執行頻率 | 每次 push | Nightly 或每次 prompt / 模型變更 |
| 結果穩定性 | 完全可重現 | 統計穩定;n=5 取平均 |

## 45.3 Eval Set 的建立與治理

Eval Set 是 Agentic QA 的核心資產。**它的價值跟你產品的程式碼一樣高,有時候更高**(因為換一個基礎模型,程式碼可能要全改,但 Eval Set 仍然有效)。

### 45.3.1 Eval Set 的三種來源

1. **黃金樣本(Golden Set)**:由領域專家手寫,代表「我們希望系統如何回應」。50–200 條起步。每條記載輸入、期望輸出特徵、評分維度。
2. **生產樣本(Production Drawn)**:從真實使用記錄中抽樣,過濾敏感資訊,人工標註。能反映真實使用模式。
3. **對抗樣本(Adversarial)**:由 Red Team(人或另一個 Agent)生成,專門找系統會失敗的邊界。見 B.5 節。

### 45.3.2 Eval Set 不是寫一次就放著

Eval Set 必須:
- **版本化**(放進 git)
- **打標籤**(category、difficulty、source)
- **定期審視**(每季一次,刪除過時、補入新型態)
- **有 owner**(不是 QA 一個人,是整個產品團隊共擔)

## 45.4 LLM-as-a-Judge:用大象來秤大象

人工評分跟不上 Eval 規模。所以業界這兩年廣泛使用 **LLM-as-a-Judge** — 用一個更強的模型(或用一個專門 prompt 過的同模型)來評分另一個模型的輸出。

### 45.4.1 一個能用的 Judge 架構

```
[受測 Agent]    →   [回應 R]
                       ↓
[原始 Query Q] →   [Judge LLM]   ←  [Evaluation Rubric]  ← [Reference Answer (optional)]
                       ↓
                  [維度評分 + 解釋]
                       ↓
                  [聚合分數]
```

**關鍵設計選擇:**

- **Pairwise vs Pointwise**。早期業界偏好 Pairwise(A 跟 B 哪個比較好),因為比較相對排序比給絕對分數穩定。但隨著模型變強,Pointwise(直接給分)也越來越可用,且更容易視覺化追蹤。
- **多維度比單一分數穩定**。不要問 Judge 「這個回答好不好(0–10)」,問它「正確性、相關性、完整性、安全性各打幾分」。混合單一分數會把所有缺陷平均掉。
- **要求引用 + 解釋**。Judge 必須說明為什麼給這個分數。沒有解釋的分數都應該不採信。

### 45.4.2 Judge 的常見偏誤(這是最容易被忽略的)

- **Position Bias**:Judge 系統性地偏向第一個 / 第二個答案。解法:每次評估都隨機交換 A/B 順序,n 次取平均。
- **Verbosity Bias**:Judge 偏好更長的答案。解法:在 rubric 明確要求懲罰冗長。
- **Self-preference**:Judge 偏好跟自己模型風格相似的答案。解法:用不同模型家族交叉驗證(GPT 受測就用 Claude 當 Judge,反之亦然)。
- **Sycophancy**:Judge 容易被 prompt 中的暗示帶風向。解法:Judge prompt 中刻意不透露受測者身份與意圖。

## 45.5 Red Teaming:把破壞當成例行公事

紅隊測試在資安領域行之有年。對 Agent 系統而言,紅隊有兩個目的:**找到讓 Agent 行為失控的 prompt**、**發現 Agent 工具誤用的路徑**。

### 45.5.1 紅隊的三個層次

| 層次 | 對抗目標 | 典型攻擊 |
|---|---|---|
| **Prompt 層** | LLM 本身 | Jailbreak、Prompt Injection、Roleplay 繞過 |
| **Agent 層** | 整個 Agent 行為 | Tool Misuse、Loop 攻擊、Context Bloat |
| **System 層** | 整套系統(Agent + 後端) | 資料外洩、權限提升、租戶汙染、資源耗盡 |

### 45.5.2 自動化紅隊的雙 Agent 架構

```
[Attacker Agent] → 生成攻擊 prompt → [受測 Agent] → 回應
       ↑                                              ↓
       └──── 學習什麼類型有效 ←── [Judge: 攻擊成功了嗎?]
```

把紅隊當成另一個 Agent 來訓練,讓它自己學會攻擊向量。每次被防住,就調整策略。這套循環跑一夜可以產生數千個高品質對抗樣本,直接餵回 Eval Set。

業界 2025–2026 比較成熟的工具:**PyRIT**(微軟)、**garak**(英國 AISI)、**Promptfoo Red Team**、**Lakera Red Teamer**。我自己會自寫一個薄層在這些之上,因為每個產品的攻擊面都不一樣。

## 45.6 Behavior Drift:沒人改 prompt,但行為變了

這是 Agent 系統最詭異的失敗模式。

**情境**:你三個月前部署了一個運作良好的客服 Agent。沒人改 prompt、沒人改模型、沒人改工具。但客服主管反映最近 Agent 開始傲慢,還會主動提供錯誤的退款政策。

### 45.6.1 為什麼會漂移?

1. **底層模型版本悄悄更新**(用了 SaaS 模型 API 的人都吃過這個虧)
2. **檢索資料庫被人加了新文件**,而且這份文件跟舊文件矛盾
3. **使用者的提問模式變了**(例如熱門事件帶來新類型問題)
4. **工具回傳格式有微調**(後端 API 加了一個欄位,看似無害,但讓 Agent 開始「探索」這個新欄位)

### 45.6.2 Drift 監測的工程實作

**Online Eval Pipeline:**
```
生產流量 → 採樣 (1–10%) → 送進 Eval 管線
                ↓
            分維度打分
                ↓
            存入時序資料庫
                ↓
            異常偵測(每日比較 7 日 / 30 日 / 90 日均值)
                ↓
            告警(分數下降超過閾值,自動建立 ticket)
```

**Drift 的三類指標必須同時監看:**

- **回答品質指標**(LLM-as-a-Judge 平均分)
- **工具使用指標**(每次對話呼叫工具的次數、失敗率、平均深度)
- **使用者反饋指標**(thumbs-up / down 比、轉接人工率、會話長度分佈)

任何一類連續三天偏離基線 > 2σ,就應該觸發人工審查。**等到使用者投訴才發現,已經晚了**。

## 45.7 把 Agentic QA 整合進 CI/CD

> 📌 [圖示建議 45-2]
> *畫一張「Agentic CI/CD」流程圖。從 Pull Request 開始,經過:Lint(秒級)→ 單元測試(分鐘級)→ Eval Mini Set(10 分鐘)→ 人工審查 → Merge → Eval Full Set(小時級)→ Canary 部署 → Online Eval(持續)→ Production。每個階段標明阻擋條件。*

| 階段 | 觸發 | 內容 | 阻擋條件 |
|---|---|---|---|
| Pre-commit | 本機 | Linter、Schema 驗證 | 任何錯誤 |
| PR Open | CI | 確定性測試 + Eval Mini(20 條) | 確定性 100%、Eval > 90% |
| Merge to main | CI | Eval Full(200+ 條) | Eval Regression > 3% 阻擋 |
| Pre-production | Manual | Red Team Suite | 任何 P0/P1 攻擊成功 |
| Canary(10% 流量) | 自動 | Online Eval | 線上分數 > 線下分數 -5% |
| Full rollout | 自動 | Drift Monitor | 三日連續 < 2σ |

- **Eval Set v1**:50 條黃金樣本,涵蓋核心使用情境。

- **CI/CD 整合方案**:寫進 GitHub Actions / GitLab CI 的 yaml 範本。

# 第 46 章｜遺留系統現代化與 AI 逆向工程

> *Brownfield Modernization with Agent-Assisted Reverse Engineering*
> 
> 已整合至：**第 46 章｜遺留系統現代化與 AI 逆向工程**（Part 07）

## 46.1 主大綱默認你在 Greenfield。但你大概率不在。

打開主大綱重讀:第 17 章談 DDD 從 Bounded Context 開始畫,第 19 章 C4 Model 從 System Context 圖開始畫,第 22 章 Event Sourcing 假設你能重新設計事件流。

**這些都是 Greenfield 視角。** 但任何在現場待過 5 年以上的工程師都知道:你接手的系統,大概率是一個 7 年前由某個離職資深工程師寫的 Java EE 應用,跑在 WebLogic 上,跟一個 PL/SQL 資料庫深度耦合,沒有單元測試,沒有文件,業務規則散落在 5000 行的 stored procedure 裡。

第 20 章提到 Strangler Fig 模式。這是現代化的「目標形狀」,但不是「行動方案」。Strangler Fig 假設你已經知道**舊系統做了什麼**。**現實是:你不知道**。原作者離職了,文件不存在,使用者只知道「按這個鈕會觸發那個流程」。

這一章談的是:**在這種狀態下,如何用 AI Agent 逆向找回知識,然後設計現代化路徑**。

## 46.2 「考古學」是新的系統分析

我把這個工作叫**程式碼考古學(Code Archaeology)**。它跟傳統 SA 的差別在於:

| 傳統 SA | 程式碼考古學 |
|---|---|
| 訪談人類 | 訪談人類 + 訪談程式碼 + 訪談資料庫 + 訪談日誌 |
| 從業務需求往下推 | 從現有系統行為往上推,再跟業務需求對校 |
| 假設原作者意圖明確 | 假設原作者意圖部分丟失,需要重建 |
| 產出 To-Be | 產出 As-Is + To-Be + Migration Path |

> 📌 [圖示建議 46-1]
> *畫一張「程式碼考古」工作流。輸入:舊原始碼、舊資料庫、舊日誌、現行使用者訪談;經過:AI Agent 靜態分析 → 業務規則提取 → As-Is 領域模型;再經過人工驗證 → 校對 → 產出 Bounded Context Map(同時涵蓋 As-Is 與 To-Be)。*

## 46.3 用 Agent 做逆向工程的四個層次

不要把 Agent 當成「貼進去問問題」的工具。它能做的事分四個層次,要分階段使用。

### Level 1:Lexical(詞彙層)
- 抽取所有類別、函式、表名、欄名清單。
- 統計呼叫頻率、依賴關係。
- 找出明顯的死碼。

**工具**:傳統靜態分析工具(SonarQube、Understand、CodeQL)即可。Agent 在這層的價值不大。

### Level 2:Structural(結構層)
- 重建模組依賴圖、類別繼承樹、資料庫 ER 圖。
- 找出循環依賴、上帝物件(God Class)、跨層耦合。
- 推測層次架構是「分層」「六角」還是「大泥球」。

**工具**:CodeQL + Agent 摘要。Agent 把工具輸出翻譯成人話與圖示。

### Level 3:Behavioral(行為層)— **Agent 真正開始發光的地方**
- 給定一個入口(例如某 REST endpoint),追蹤執行路徑。
- 從 5000 行 stored procedure 中提取出可讀的業務規則。
- 識別「看似相同但暗藏差異」的程式碼片段(常見於複製貼上演進)。
- 把日誌反推回程式碼路徑,找出真正被觸發的分支。

**工具**:Claude Code、Cursor、Antigravity、Codex CLI。需要長上下文與工具呼叫。

### Level 4:Domain(領域層)
- 從程式碼推回業務語言(Ubiquitous Language)。
- 為每一個推測出來的 Bounded Context 寫一段「考古報告」。
- 對照現行使用者訪談,標記**矛盾點**(程式碼這樣寫,但使用者描述不一樣 → 一定有故事)。

**這一層必須人類主導,Agent 輔助**。Agent 可以提案,人類必須驗證。

## 46.4 IDE / Agent 工具的取捨(2026 現實)

主大綱第 34 章提到了 Context-Driven Engineering 與工具,但沒有針對「逆向工程」場景做選型。這裡補上:

| 工具 | 上下文視窗 | Brownfield 適合度 | 我的取捨 |
|---|---|---|---|
| **Google Antigravity**(Gemini 3.1 Pro) | 200 萬 token | ★★★★★ 大型 codebase 整體理解 | 可以一口氣吃下整個中型 monorepo,但目前 production 穩定性仍差,Google 自己內部都不全用。**用它做分析,不用它做生產修改**。 |
| **Cursor**(Claude Sonnet 4.6 / GPT-5.1) | ~200K | ★★★★ 精準局部修改 | SOC 2 + GDPR 合規通過,適合在嚴肅環境做小範圍修改。Shadow workspace 是 Brownfield 救命特性。 |

2. **Claude Code** — 主力編碼,跑深度逆向工程,提取業務規則,寫進文件。Skills 設計成「逆向某個 stored procedure 並產出 PRD」這類。
3. **Cursor** — 進入修改階段時做精準 refactor,Shadow workspace 確保不污染主分支。

## 46.5 從 As-Is 到 To-Be 的安全平滑遷移

這是最容易翻車的環節。一旦你提取出 As-Is 領域模型,開始畫 To-Be,**80% 的失敗在這裡發生**:把 To-Be 畫得太理想、跟 As-Is 不接、導致遷移路徑根本走不通。

### 46.5.1 三條黃金規則

**規則一:To-Be 必須能用 As-Is 的語言被解釋。** 如果 To-Be 引入了 As-Is 完全沒有的概念(例如新的 Bounded Context),你必須能在現有業務語言中為它找到對應。否則使用者訓練成本會殺死整個專案。

**規則二:任何 To-Be 模組必須有「並行運轉一個月」的計畫。** 新模組接管舊模組之前,讓兩者並行跑、輸出比對。Stripe、GitLab、Shopify 在大型遷移上都用這套。

**規則三:Strangler Fig 必須有「我可以中途停下來」的設計。** 如果你做到一半發現方向錯了,要能停在中間狀態繼續運轉,而不是「全有或全無」。這意味著舊新模組之間必須維持雙向相容。

### 46.5.2 一個典型的遷移迭代

```
迭代 N 開始
├─ Step 1: 選擇一個邊界明確的舊模組(用 As-Is 領域模型挑)
├─ Step 2: 寫契約測試(以舊模組行為為基準)
├─ Step 3: 用 Agent 生成 To-Be 模組骨架
├─ Step 4: 並行運轉(舊模組仍是 source of truth)
├─ Step 5: 比對輸出,差異分類:
│    ├─ 舊模組的 bug → 為 To-Be 寫對的版本,留 ADR
│    ├─ To-Be 的 bug → 修
│    └─ 業務歧義(沒人記得當初為什麼) → 召集利害關係人決議
├─ Step 6: 切換流量(從 1% canary 開始,逐週加碼)
└─ Step 7: 退役舊模組
```

每一個迭代產出一份 ADR + 一份「考古報告」,慢慢累積整個系統的知識資產。

## 46.6 踩坑清單

- **不要相信 Agent 的初次摘要**。給它一個 2 萬行的 service,它會自信地告訴你業務邏輯。請對重要環節人工驗證至少 3 次。
- **小心「幻覺出來的 API」**。Agent 有時會發明不存在的函式呼叫。寫程式碼前必須驗證真實存在。
- **不要讓 Agent 自己決定刪除程式碼**。死碼分析永遠假設「我可能漏看了某條觸發路徑」。要刪,必須有人簽字。
- **舊系統的權限模型常常是隱含的**。例如「這個欄位只有特定 role 看得到,但檢查邏輯散在 5 個地方」。Agent 容易遺漏,要特別審視。
- **資料庫 trigger 是地雷區**。Stored procedure 之外,trigger 是另一個常被遺忘的業務規則藏身處。
- **跨系統整合常常沒文件**。看似獨立的舊系統,可能有 5 個爬蟲、3 個 cron job 在背後共享資料。Agent 看不到這些,要靠人類訪談 + 日誌分析。

## 46.7 設計交付清單

- **As-Is Bounded Context Map**:從程式碼考古推回的領域模型。
- **業務規則目錄**(Business Rules Catalog):每條規則記載出處、推導理由、置信度。
- **遷移依賴圖**:模組之間的拆解順序與依賴。
- **遷移 ADR 集**:每一個重要決策有理由。
- **並行運轉比對報表**:舊新差異分類與處置紀錄。

# 第 40 章｜Multi-Agent 共識、狀態與衝突解決

> *Multi-Agent State Management & Consensus*
> 
> 已整合至：**第 40 章｜Multi-Agent 共識、狀態與衝突解決**（Part 07）

## 40.1 主大綱第 36 章畫了拓樸,但沒談「他們不同意怎麼辦」

主大綱第 36 章列出了 Multi-Agent 的五種拓樸(Single + Tools、Supervisor-Worker、Hierarchical、Network、Distributed Collaboration)。第 21–22 章談了傳統微服務的分散式交易(Saga、Outbox)。

**但兩者沒有交叉。**

問題是這樣的:**當兩個自主 Agent 在沒有中心化資料庫的前提下,需要對某件事達成共識(例如:這筆訂單的責任歸誰、這次資源排程怎麼分、這個會議結論是什麼),你怎麼辦?**

這不是學術問題。它每天在以下場景發生:

- 採購 Agent 與庫存 Agent 對「一個 SKU 還剩幾件可用」有不同看法。
- 兩個排程 Agent 同時試圖把同一個工程師排到兩個會議。
- 多個分析 Agent 對同一份財報得出衝突的結論,主管 Agent 必須整合。
- Code Review Agent 與 Security Agent 對同一段程式碼一個說 LGTM 一個說 Block。

傳統 Saga 解決的是「跨服務交易的補償」,但它**默認最終誰的版本是對的**(通常是 orchestrator 的版本)。Agent 系統裡沒有這種「天然權威」。

> 📌 [圖示建議 40-1]
> *畫一張對比圖:左邊是「傳統微服務 Saga」,有明確的 orchestrator 與補償流;右邊是「Multi-Agent 衝突」,沒有 orchestrator,但有多個 Agent 各自有意見,中間有個問號標明「誰是對的?」*

## 40.2 三種共識需求,不要混為一談

第一步是**承認你需要的不是「一種共識」**。把它分成三類:

| 共識類型 | 例子 | 傳統對應 |
|---|---|---|
| **資料共識**(誰的數值是對的) | 「庫存到底剩幾件」 | Replicated State Machine、Raft |
| **權威共識**(誰說了算) | 「這次申請該由 A 還是 B 簽核」 | RBAC + Workflow Engine |
| **意義共識**(這代表什麼) | 「這封 email 是 P0 還是 P2」 | (傳統沒對應,人類來判斷) |

主大綱第 21 章的工具(Saga、Outbox)解決的是「資料共識」。Agent 系統最常需要的反而是**意義共識** — 這是傳統分散式系統幾乎沒處理過的問題。

## 40.3 解決方案 1:重建「中央但不單點」的共識層

最務實的解法,通常不是讓 Agent 自己達成共識,而是**設計一個受信任的共識層**。它不是一個 Agent,是一個**協議**。

### 40.3.1 Event Log as Truth

把所有 Agent 的所有重要動作寫進一條 append-only event log(Kafka、EventStoreDB、自建),log 本身就是 source of truth。Agent 之間的「認知」由 log 決定。

```
Agent A: 「我認為訂單 #123 已付款」
        → emit event: OrderPaid(123, t=10:00, by=A)
Agent B: 「我認為訂單 #123 還沒付款」
        → 讀 log,看到 A 的事件
        → 決定信任 vs 質疑
        → emit event: PaymentDisputed(123, t=10:01, by=B, reason=...)
        
Resolver Agent / Human: 讀 log,看到衝突
        → emit event: PaymentResolved(123, t=10:05, decision=A_correct)
```

這套等於把 Event Sourcing 升級為**多 Agent 的共享記憶與爭議仲裁機制**。每個 Agent 只能對 log 提出主張(propose),不能直接改變狀態。

**這個模式我用得最多。它把問題從「分散式共識」轉化為「事件序列推論」,難度大幅下降。**

### 40.3.2 兩階段共識(2-Phase Agreement)

對於需要強一致性的場景(資金、庫存):

```
Phase 1 (Propose): 
  Agent A: 「我打算扣 5 件庫存」→ Coordinator
  Coordinator → 廣播給其他相關 Agent
  其他 Agent: 「ACK / NACK + 理由」
  
Phase 2 (Commit):
  若全部 ACK → Coordinator 寫入 → 通知全體
  若有 NACK → 啟動衝突解決(見 D.4)
```

這在 Agent 世界跟資料庫的 2PC 在概念上一樣,但有兩個關鍵差異:
- **Phase 1 的 NACK 可以包含「為什麼不同意」的推理**,而不只是 yes/no。
- **Phase 2 失敗的補償可以是 Agent 自己決定的**(例如「用另一個策略再嘗試」),不像資料庫只能 rollback。

## 40.4 解決方案 2:衝突解決協議(Conflict Resolution Protocols)

當 Agent 之間真的需要協商(沒有共識層,或共識層也搞不定的意義衝突),你需要一個**明確的協商協議**。

### 40.4.1 為什麼不能讓 Agent 自由聊?

直覺上會讓兩個 Agent 「自由對話直到達成共識」。**這是一個陷阱**。

- **不收斂**:它們可能無限對話,各自舔尾巴。
- **被誤導**:一個有缺陷的 Agent 可能說服另一個。
- **代幣爆炸**:每輪對話的 context 越來越長,成本指數爆炸。

務實做法:**有結構的協商**。

### 40.4.2 一個我用過的協議:Bounded Negotiation

```
Round 1 (Position):
  每個 Agent 必須在 N 字以內陳述立場 + 證據

Round 2 (Question):
  每個 Agent 對其他 Agent 提一個問題

Round 3 (Update):
  每個 Agent 在新資訊基礎上,決定維持或修正立場

Decision Gate:
  收斂(全部立場一致) → Done
  不收斂 → 升級到 Resolver Agent / Human
  
最大輪次:3 輪。超過就強制升級。
```

這套可以在各種拓樸下實作(Network、Hierarchical、Supervisor)。關鍵是:**有明確的退出條件,不能無限循環**。

### 40.4.3 仲裁者(Arbiter / Resolver)的設計

當 Bounded Negotiation 不收斂,需要一個 Arbiter。Arbiter 通常是:
- **更強模型** — 用 GPT-5 / Claude Opus 仲裁 Sonnet 之間的爭議。
- **領域專家 Agent** — 在採購衝突中,引入「採購 SOP Agent」做仲裁,它有完整的政策知識。
- **人類** — 最後手段,但要設計成易接入(把所有 context 整理成一頁摘要,人類點 A/B 即可)。

> 📌 [圖示建議 40-2]
> *畫一張「Bounded Negotiation 流程圖」,三輪 Position-Question-Update,後面接 Decision Gate,分岔出「達成共識」與「升級到 Arbiter」兩條路。*

## 40.5 分散式狀態機(Distributed State Machine)

當 Agent 之間有複雜的多步協作(比 Saga 更複雜),你需要顯式的**分散式狀態機**。

### 40.5.1 為什麼狀態機重要?

人類傾向把 Agent 行為描述成「自然語言流程」。**這是調試與審計的災難**。把它寫成狀態機,你立刻獲得:

- **明確的狀態列舉**(一共有哪些可能狀態)
- **明確的轉換條件**(從 A 狀態到 B 狀態需要什麼)
- **可驗證**(可以證明不會卡死、不會回到非法狀態)
- **可審計**(每次轉換都有 trace)

### 40.5.2 工具選擇

- **LangGraph**:LangChain 系列,以圖建模 Agent 流程。事實上的入門選擇。
- **Temporal**:不是 Agent 專用,但是「持久化工作流」的標竿,Agent 與傳統工作流混合的最佳載體。
- **AWS Step Functions / Azure Durable Functions**:雲廠對應品。
- **自建 + Redis Streams**:小團隊預算有限時的務實選擇。

### 40.5.3 一個關鍵的反模式:不要讓 LLM 「自由決定」狀態轉換

有些教學會教你:讓 LLM 看當前 context,「想想下一步要去哪」。**請避開這種模式**。LLM 應該決定「在當前狀態下要採取什麼動作」,但「動作觸發後系統處於什麼狀態」必須由確定性的轉換函數決定。

```
正確:
  state = "pending_review"
  llm_action = llm.decide(state, context)  # → "approve" / "request_change"
  new_state = transition_table[state][llm_action]  # 確定性

錯誤:
  llm_decision = llm.decide(state, context)  # → "ok 我覺得我們現在進入 finalized 狀態"
  state = llm_decision  # 把狀態本身交給 LLM 決定
```

## 40.6 失敗模式速查

| 症狀 | 根因 | 解法 |
|---|---|---|
| Agent 之間無限對話 | 沒設定退出條件 | Bounded Negotiation + 強制 Arbiter 升級 |
| 兩個 Agent 都堅持自己對 | 缺乏共享 source of truth | 引入 Event Log |
| 共識達成但結果是錯的 | Sycophancy(都說好聽話互相認同) | 引入專業 Critic Agent;不同模型家族交叉 |
| 小衝突拖垮整個系統 | 沒有「快速失敗」路徑 | 轉換失敗 N 次直接升級;不要無止境補償 |
| 共識日誌爆炸 | 把太瑣碎的事都寫進共識層 | 只寫業務級事件;低層細節走自己的 trace |

## 40.7 設計交付清單

- **共識需求分類表**:你系統中需要哪幾類共識(資料 / 權威 / 意義)
- **Event Log Schema**:Agent 主張、爭議、解決三類事件的格式
- **Negotiation Protocol**:你採用的協商協議的明確規格(輪數、退出條件、Arbiter)
- **State Machine 圖**:Multi-Agent 協作的所有狀態與轉換
- **失敗劇本**:每一類失敗模式的處理 SOP

# 第 28 章｜Compliance by Design ⸺ AI 合規架構

> *Designing for Compliance from Day Zero*
> 
> 已整合至：**第 28 章｜Compliance by Design ⸺ AI 合規架構**（Part 05）

## 28.1 2026 年 8 月 2 日

這個日期不是隨便挑的。**2026/8/2 是 EU AI Act 高風險 AI 系統合規的全面執行日**(Annex III 系統)。這意味著到那天為止,你的高風險 AI 系統必須:

- 完成符合性評估(Conformity Assessment)
- 完成技術文件(Technical Documentation)
- 取得 CE 標誌(CE Marking)
- 在歐盟資料庫註冊(EU Database Registration)
- 建立風險管理系統與品質管理系統(QMS)
- 具備事件回報機制(72 小時 / 15 日 雙時程)

罰則上限是 **3,500 萬歐元或全球年營收的 7%**(以較高者計)。GDPR 在 2025–2026 仍同時適用,違反個資相關義務可疊加處罰至 **2,000 萬歐元或全球年營收的 4%**。

主大綱第 25 章把「對抗性安全」(Prompt Injection、Tool Misuse 等)作為主軸。**但合規(Compliance)不是安全(Security)**。安全防的是「壞人」,合規防的是「自己也可能犯的錯」。

## 28.2 三個必須在系統設計層面解決的合規挑戰

### 28.2.1 挑戰一:RAG 系統的細粒度存取控制(RBAC × Row-Level)

典型場景:你做了一個企業內部的 RAG 知識庫。HR 文件、財務文件、研發文件、客戶資料都進去了。**問題:檢索器(Retriever)在語意搜尋時,根本不知道「這個提問者是誰、能看哪些文件」。**

最常見但**錯誤**的做法:在 LLM 的 prompt 裡加一句「請不要把 HR 文件給研發部門看」。**這是字面意義上的 prompt injection 自殺**。LLM 會被各種繞過技巧攻破。

正確的做法是:**把存取控制做在檢索層,讓 LLM 看不到不該看的內容**。

```
Query → 取得使用者身份 → 從目錄服務取得 role 與屬性
                              ↓
                          建立 ACL 過濾器(metadata filter)
                              ↓
                       向量檢索 + ACL 過濾(同時)
                              ↓
                      只回傳「使用者有權看的」chunk
                              ↓
                          送進 LLM 生成回答
```

**關鍵是「同時」**。先檢索後過濾(Post-filter)會洩漏資訊(從結果數量推測存在哪些文件)。要前融合過濾(Pre-filter at index level)。

支援這個能力的向量資料庫:Qdrant(Payload Filter)、Weaviate(Where Filter)、Pinecone(Metadata Filter)、pgvector + 行級安全(RLS)。pgvector + Postgres RLS 是合規場景的隱藏寶藏 — Postgres RLS 是 RBAC 細粒度控制的成熟方案,加上 pgvector 直接同時得到語意搜尋與行級權限。

### 28.2.2 挑戰二:動態資料遮罩(Dynamic Data Masking)

不是所有有權看文件的人,都有權看文件**全部內容**。

例如:客服 Agent 可以看到客戶的訂單歷史(因為要服務),但**不該看到客戶的完整身分證號**(只能看後 4 碼)。傳統做法是在資料倉庫層做永久遮罩。但 RAG 系統的問題是:**遮罩必須在檢索時動態執行**,因為同一份原始文件,給不同使用者看的版本不一樣。

**動態遮罩的三層方案:**

| 層次 | 實作位置 | 適用 |
|---|---|---|
| **儲存層遮罩** | Postgres Row-Level Security + 視圖 | 結構化資料 |
| **索引層遮罩** | 在索引時對敏感欄位 hash / tokenize | 預知敏感欄位、不會變 |
| **檢索層遮罩** | 取出 chunk 後、進 LLM 前,用正則 / NER 把 PII 替換成 placeholder | 非結構化文件、規則複雜 |

實務上**通常三者並用**。儲存層擋大規模洩漏、檢索層擋語意搜尋洩漏、索引層做基礎處理。

### 28.2.3 挑戰三:多租戶上下文隔離(Tenant Isolation)

如果你的 SaaS 服務多個客戶,**最致命的合規災難就是「A 客戶的問題,Agent 答出 B 客戶的資料」**。

這不只是 bug,是**合約違反 + 個資法違反 + 信任崩潰**的三重打擊。

四個層次都必須隔離:

1. **儲存隔離**:不同租戶資料用不同的 schema / database / 索引,而不是共用一張表加 `tenant_id` 欄位(後者太容易因為一行 bug 就洩漏)。
2. **計算隔離**:Agent 工作流的 context、cache、向量空間都按 tenant 分。Redis 分庫、向量資料庫的 collection 分租戶。
3. **模型隔離**:同一個 LLM 處理不同租戶的請求時,**不能跨請求保留上下文**。每次請求 context 必須完全重置。Memory 元件必須帶 tenant_id 索引。
4. **遙測隔離**:OTel trace、logs、metrics 都帶 tenant_id 標籤;儀表板按租戶切;故障分析時不會把 A 的資料當作 B 的。

> 📌 [圖示建議 28-2]
> *畫一個四層的租戶隔離架構圖。橫軸是租戶 A、B、C;縱軸是儲存、計算、模型、遙測。每個交點標明隔離手段。標出常見洩漏路徑(例如共用 prompt cache、共用 vector index)用紅線示警。*

## 28.3 EU AI Act 合規的工程映射

很多人讀法規讀到頭痛,因為法律語言跟工程語言對不上。這張表是我自己用的對映:

| AI Act 條文 | 工程意義 | 你應該做的事 |
|---|---|---|
| Art. 9 風險管理系統 | 系統性辨識、緩解、監控風險 | 建立 RMS 文件 + 季度更新流程 |
| Art. 10 資料治理 | 訓練 / 驗證 / 測試資料的代表性與品質 | 資料卡(Datasheet for Datasets);偏誤評估 |
| Art. 11 技術文件 | Annex IV 規定的 9 大類文件 | 用 ADR + Model Card + System Card 組合產出 |
| Art. 12 紀錄保存 | 自動紀錄系統行為 | 結構化日誌 + 不可篡改保存(WORM) |
| Art. 13 透明度與資訊提供 | 使用者知道在跟 AI 互動,且能理解輸出 | UI 上明示;提供解釋介面 |
| Art. 14 人類監督 | 高風險決策必須有人類介入點 | Human-in-the-Loop 設計;一鍵取消按鈕 |
| Art. 15 準確性、強健性、資安 | 量化的品質保證 | Eval Set + Drift Monitor + Red Team(見第 45 章) |
| Art. 17 品質管理系統 | 全生命週期的品質流程 | ISO/IEC 42001 對齊 |
| Art. 26(4) 部署者監控 | 持續監測,72h 重大事件 / 15 日嚴重事件回報 | Incident Response Plan + 法務通報自動化 |
| Art. 27 基本權利影響評估(FRIA) | 高風險部署前的影響評估 | FRIA 模板;每個 high-risk use case 一份 |

## 28.4 DPIA × FRIA:不是兩份,是一份

GDPR 要求 DPIA(資料保護影響評估),AI Act 要求 FRIA(基本權利影響評估)。實務上**寫成兩份是浪費**。

我採用的整合範本(七節):

```
1. 系統描述(對應 DPIA Art.35.7(a) + FRIA Art.27(1)(a))
2. 必要性與比例性(DPIA Art.35.7(b))
3. 資料流與處理活動(DPIA 主要;FRIA 也需要)
4. 風險辨識
   ├─ 個資風險(DPIA)
   └─ 基本權利風險(FRIA)— 歧視、自主性、隱私之外的權利
5. 緩解措施(技術 + 組織)
6. 殘餘風險與接受度
7. 諮詢紀錄(資料保護長 / 利害關係人 / 影響族群代表)
```

這份文件**必須在系統上線前完成,且會被監管機關要求調閱**。

## 28.5 不要等到法務找你才開始

合規最常見的失敗模式不是「做不到」,而是「太晚才開始做」。

**在 PoC 階段就應該做的事:**

- 在 PRD / SRS 加一節「合規分類」(這是不是 AI Act high-risk?是不是 GDPR high-risk processing?)
- 資料攝入流程從第一天就保留出處(Provenance)
- Eval Set 從第一天就包含「歧視 / 偏誤」維度
- ADR 範本加一欄「合規影響」

**在 MVP 階段就應該完成的事:**

- 第一版 DPIA × FRIA
- 第一版 Model Card / System Card
- 事件回報機制接到法務通報

**在 GA 之前必須完成的事:**

- 完整 Annex IV 技術文件
- CE Marking 流程啟動
- 第三方稽核(若適用)
- AI Office 註冊

- **合規分類表**:你的系統落在 AI Act 哪一個風險等級?涉及哪些 GDPR 條文?
- **DPIA × FRIA 整合報告**
- **Model Card / System Card / Datasheet**
- **RBAC × Row-Level 存取控制設計**

- **多租戶隔離架構文件**
- **事件回報 SOP**(包含 72h / 15 日時程)

# 第 17 章｜多模態與對話式互動的系統分析

> *Multimodal & Conversational UX (CUX) Systems Analysis*
> 
> 已整合至：**第 17 章｜多模態與對話式互動的系統分析**（Part 03）

## 17.1 古典分析語彙的破口

主大綱第 16 章談 UI/UX 系統觀,提到 Information Architecture、Atomic Design、Generative UI。第 9 章談流程模型(BPMN、活動圖、狀態機)。

這些工具的共同假設是:**互動是離散的、有先後的、可編號的**。使用者點一個按鈕、跳到下一個畫面、輸入欄位、按確定。

但 2025–2026 的人機互動正在發生根本轉移:

- **語音優先(Voice-First)**:中國 / 日本 / 美國的智慧音箱與車載助理已經是主介面。
- **聊天優先(Chat-First)**:絕大多數新世代企業 SaaS 加上 Copilot,Copilot 變成首要入口。
- **相機作為感應器(Camera-as-Sensor)**:即時視覺串流(直播零售、AR 維修、無人巡檢)變成持續輸入,而不是「拍一張照」。
- **空間運算(Spatial Computing)**:Apple Vision Pro、Meta Quest 把使用者放在三維空間中,目光、手勢、身體姿態成為互動向量。

這些互動有三個共同特徵,讓古典分析工具破功:

1. **連續而非離散**:沒有清晰的「步驟 1 → 步驟 2」界線。
2. **多軌平行**:使用者可以同時說話 + 比手勢 + 看著某個物件,三個訊號同時產生。
3. **充滿中斷**:語音被環境噪音打斷、攝影機被遮蔽、視線飄走 — 系統必須能在「資訊缺失」狀態下優雅運作。

> *畫一張對比圖:左邊「古典 GUI 互動」(離散步驟、單軌、線性),用 BPMN 或活動圖表現;右邊「CUX / 多模態互動」(連續、多軌平行、頻繁中斷),用一條時間軸 + 三個並行訊號軌道(語音、視覺、姿態)+ 中斷標記表現。*

## 17.2 為什麼 BPMN 在這裡破功

我用一個具體例子說明。你正在設計一個「居家視訊看護」系統:長者在客廳,系統用攝影機 + 語音助理持續監看。場景:**長者跌倒了**。

用 BPMN 畫這個流程,你會這樣寫:

```
攝影機捕捉到跌倒 → 系統告警 → 語音問「您還好嗎?」→ 長者回應 → 判斷是否報警
```

**這是錯的。**真實情況是:

- 攝影機可能誤判(影子、坐姿改變)
- 長者可能正在睡著沒回應
- 語音問問題的當下,長者可能已經昏迷
- 鄰居敲門的聲音可能被誤認為「跌倒撞擊聲」
- 長者可能說「我沒事」但實際語調顫抖

BPMN 的活動框與決策菱形,完全表達不出**「機率性訊號」「持續推論」「跨模態融合」「不確定下的決策」**這四件事。它的語意太離散、太線性、太確定。

我們需要一套新的分析語彙。

## 17.3 替代分析框架:Signal-Decision-Confidence(SDC)模型

我自己用的分析框架,從訊號處理 + 決策論借來:

```
[Continuous Signals 連續訊號]
    ├─ 語音流 (audio stream)
    ├─ 視覺流 (video frames)
    └─ 環境訊號 (環境音、IoT sensor)
              ↓
[Feature Extraction 特徵抽取]
    ├─ 跌倒偵測模型輸出
    ├─ 語音情緒分析
    └─ 物件辨識
              ↓
[Multi-Modal Fusion 多模態融合]  ← 把多軌訊號融合成一個帶信心度的判斷
              ↓
[Decision with Confidence 帶信心度的決策]
    ├─ 信心 > 0.95 → 直接行動(撥打 119)
    ├─ 信心 0.7–0.95 → 二次確認(語音問候)
    ├─ 信心 0.4–0.7 → 留意 + 升級監看
    └─ 信心 < 0.4 → 忽略
              ↓
[Action with Reversibility 可逆 / 不可逆動作]
    ├─ 可逆(亮個燈、改變介面) → 容錯高
    └─ 不可逆(撥打 119) → 容錯極低,需多重驗證
```

這個框架對應到 SA 的工件,是新的:

| SDC 元素 | 對應的分析產出物 |
|---|---|
| Continuous Signals | **訊號目錄(Signal Catalog)** — 列出所有持續訊號的格式、頻率、品質 |
| Feature Extraction | **模型契約(Model Contracts)** — 每個推論模型的輸入、輸出、信心度語意 |
| Multi-Modal Fusion | **融合規則表(Fusion Rule Table)** — 多訊號的權重、衝突解決、降級策略 |
| Decision with Confidence | **信心度決策樹(Confidence Decision Tree)** — 不同信心區間的行動邏輯 |
| Action with Reversibility | **不可逆動作清單(Irreversible Action List)** — 哪些動作必須極高信心 + 多重驗證 |

> 📌 [圖示建議 17-2]
> *繪製一張完整的 SDC 模型圖,包含五層,並用居家看護或客服語音助理為實例,具體填入內容。*

## 17.4 對話式介面(CUX)的特殊分析語彙

對話流不像表單,它沒有固定欄位順序。但它仍然有結構,只是這個結構是**意圖網絡(Intent Graph)**而非流程圖。

### 17.4.1 意圖網絡的三個層次

```
Layer 1: Intent (意圖)
    例:「我要訂位」「我要查詢」「我要取消」
    
Layer 2: Slots (槽位)
    每個 Intent 需要哪些資訊才能完成
    例:訂位需要 [日期, 時間, 人數, 偏好]
    
Layer 3: Confirmation (確認)
    系統如何讓使用者確認決策
    例:覆述 / 部份覆述 / 預覽 / 直接確認
```

### 17.4.2 對話流的五種典型路徑

不要試圖用單一流程圖捕捉對話。用「五條典型路徑」+ 「邊角案例集」:

| 路徑 | 特徵 | 設計重點 |
|---|---|---|
| **Happy Path** | 使用者一次給齊資訊 | 只需確認,不要囉嗦 |
| **Slot Filling** | 一次給部份,系統逐步問 | 問題順序按重要性 |
| **Disambiguation** | 使用者意圖模糊 | 提供 2–3 個明確選項 |
| **Correction** | 使用者改主意 | 必須能無痛回退 |
| **Bailout** | 使用者放棄 / 升級到人 | 摩擦低、不羞辱使用者 |

每條路徑都應該有獨立的對話腳本與評估標準。

### 17.4.3 邊界案例必修

- **沉默處理**:使用者說一半停下來,系統等多久才開口?(經驗值:語音場景 3–5 秒,文字場景無限等待)
- **打斷處理**(Barge-in):使用者在系統說話時直接打斷,系統必須立刻停。
- **錯誤恢復**:語音辨識 ASR 錯了,系統如何優雅修復?重複問可能讓使用者爆炸。
- **多人對話**(Cocktail Party):多人同時說話,系統聽誰?這在車載與會議場景每天遇到。
- **情緒升級**:使用者開始焦躁,系統必須能識別並降階(降速、改詞、轉人工)。

## 17.5 視覺即時串流(Camera-as-Sensor)的系統分析

當攝影機從「拍一張照」變成「持續輸入」,SA 工件也要對應改變。

### 17.5.1 串流分析的「採樣決策圖」

```
原始視訊流 (例如 30 fps)
        ↓
[Frame Sampler] ← 採樣策略(以下三選一或混搭)
    │
    ├─ Time-based: 每 N 秒一張
    ├─ Event-based: 偵測到變化才取
    └─ Hybrid: 平時 5 秒一張,偵測到事件後加密到 1 秒一張
        ↓
[Per-Frame Inference] ← 物件偵測 / 動作辨識 / 異常偵測
        ↓
[Temporal Aggregation] ← 跨 frame 整合(避免單張 false positive)
        ↓
[Decision]
```

兩個踩坑:
- **不要把 30fps 整段送進雲端做推論** — 頻寬與成本會爆炸。一定要在 Edge 做 Frame Sampler(回看第 26 章)。
- **不要相信單張 frame 的判斷** — 用 Temporal Aggregation。例如「連續 3 張都偵測到跌倒姿態才告警」。

### 17.5.2 視覺即時串流的隱私

這部分跟第 28 章 緊密相關:**攝影機是個資災區**。

- 邊緣處理(Edge-Inference):只把判斷結果上傳,原始影像不上傳。
- 即時去識別化(Real-time Anonymization):上傳前打馬賽克 / 替換臉部。
- 同意管理:被拍到的人是否同意被處理?(尤其是公共場域)
- 留存最小化:推論完即刪,不要為了「以防萬一」留 30 天。

## 17.6 設計交付清單

- **互動模式分類**:你的系統有哪些互動模式(GUI / Voice / Chat / Vision / Spatial)?
- **訊號目錄(Signal Catalog)**:每個連續訊號的規格。
- **模型契約集合**:每個推論模型的輸入輸出。
- **融合規則表**。
- **信心度決策樹**:不同信心區間的行動。
- **不可逆動作清單**。
- **對話流五路徑 + 邊角案例集**(僅 CUX 場景)。
- **採樣決策圖 + 串流隱私設計**(僅 Vision 場景)。

## G.1 上線一年後,沒有人知道系統現在的狀態

我見過一個「設計得非常彈性」的企業平台。架構師很驕傲地說:「我們的系統高度可配置,幾乎所有行為都能透過 feature flag 和 config 調整,不需要改程式碼。」

十四個月後,我被請去做健檢。我問了幾個問題:

- **「現在有幾個 feature flag 是開著的?」** 沒有人知道確切數字,大概是「幾十個?可能超過一百?」
- **「這個 flag 是什麼時候開的、為什麼開的?」** 原始開發者已離職,沒有人記得。
- **「如果我把這個 config 從 A 改成 B,會影響哪些地方?」** 沒有人能在三十分鐘內回答。
- **「我們有多少種有效的配置組合?」** 沒有人算過。(答案:理論上是 2 的 N 次方,N = flag 數量)

這個系統不是高度彈性,它是高度**不可治理**。每一個 feature flag 都是一條可能的行為分支;每一個未記錄的 config 改動都是一顆不定時炸彈。他們花在「搞清楚系統現在怎麼設定的」的時間,比花在「開發新功能」的時間還多。

主大綱第 21–22 章談了分散式系統的狀態一致性,第 25 章談了對抗性安全。**但系統分析很少觸碰治理(Governance)這個議題。治理不是管理學的詞彙,它是一個工程問題:你怎麼確保「允許改變的部分」被改變時,系統行為仍然可預測、可追蹤、可復原?**

> 📌 [圖示建議 G-1]
> *畫一條時間軸:Day 0(系統上線)→ Month 3(第一個緊急 hotfix config)→ Month 6(原始設計師離職)→ Month 9(沒人知道這個 flag 在做什麼)→ Month 14(健檢時的困境)。每個節點標注「複雜度增量」,最後的曲線呈指數成長。*

## G.2 彈性是乘法器,它把治理缺失放大

### G.2.1 彈性的狀態爆炸

一個系統有 N 個獨立的布林 feature flag,就有 2^N 種理論上可能的配置狀態。當 N = 10,這是 1,024 種。當 N = 30,這是超過十億種。

但在實務上,「彈性」不只有布林 flag:

- **多值配置**:一個 config 有 5 個選項,就乘以 5 倍狀態空間
- **依賴關係**:flag A 只在 flag B 開著時有意義;flag C 和 flag D 不能同時開
- **環境差異**:dev / staging / prod 的 config 不一樣
- **時序依賴**:某個 config 在某個時間點是對的,在另一個時間點是錯的

每一個「彈性點」都是一個治理負債項目。設計時沒有配套的治理機制,這些負債會以指數方式複利成長。

### G.2.2 治理缺失的三種死法

```
死法一:配置漂移(Configuration Drift)
    各環境的 config 各自演化,沒有人追蹤差異
    → 「在我的機器上可以跑」的分散式版本
    → 最終:prod 行為和任何人的預期都不一樣

死法二:知識流失(Knowledge Decay)
    原始設計意圖只存在原始設計師的腦袋裡
    → 人一走,沒有人知道為什麼這樣設定
    → 最終:沒有人敢改,系統成為禁忌神器

死法三:隱性耦合(Invisible Coupling)
    config 之間有依賴關係但沒有文件記錄
    → 改一個看似無關的 flag,觸發遠端的奇怪行為
    → 最終:每次改動都是一次未爆彈拆除
```

## G.3 治理系統的五個層次

這五個層次不是獨立的選項,而是一個完整的堆疊。從下到上依序建立。

### G.3.1 層次一:機制與政策分離(Mechanism-Policy Separation)

最根本的設計原則,也是其他四個層次的基礎。

把系統切成兩層:

| 層次 | 特性 | 設計目標 | 例子 |
|---|---|---|---|
| **機制層** | 固定、通用、穩定 | 永遠不需要為了業務需求而改 | Event log 格式、Contract schema、API interface |
| **政策層** | 可配置、可版本化、可稽核 | 是唯一被允許改變的層 | 哪個角色能核准哪類請求、哪些 flag 在哪個環境開啟 |

**核心規則:複雜度只允許在政策層增長。機制層一旦設計完成,不因業務需求而改動。**

實作形式:政策層通常是版本控制下的 JSON / YAML / Rego(OPA)檔案,而不是程式碼邏輯。每一次政策變動就是一個 git commit,有時間戳、有作者、有 diff、有 review 紀錄。

```
錯誤的方式:
    if (user.role === 'admin' && feature_flag_x && env === 'prod') { ... }
    // 業務規則混在程式碼裡,無法在不改 code 的情況下修改

正確的方式:
    policy.evaluate('can_access_feature_x', { user, env })
    // 機制:evaluate() 永遠不變
    // 政策:policy 定義存在版本控制的 policy.rego,可獨立修改、測試、回滾
```

### G.3.2 層次二:黃金路徑(Golden Path / Paved Road)

**讓 80% 的使用場景走不需要行使彈性的預設路徑。**

黃金路徑不是限制,是「預鋪好的最優解」。偏離黃金路徑是被允許的,但需要付出治理成本:留下偏離理由、通過對應層級的審批、承擔後續維護責任。

```
黃金路徑示意:

標準流程 ─────────────────────────────────→ 產出
              ↑
     80% 的案例走這裡
     零配置,零治理成本

偏離點 ──→ 偏離路徑 A ──→ 需要 [記錄原因 + 審批]
         ↘ 偏離路徑 B ──→ 需要 [記錄原因 + ADR + 人工審查]
```

黃金路徑的設計原則:
- 覆蓋 80% 使用場景的預設值,不是「最彈性的預設值」
- 偏離的代價要明確標示(不是懲罰,是透明化)
- 黃金路徑本身需要定期重新評估(過時的黃金路徑比沒有更糟)

### G.3.3 層次三:複雜度預算(Complexity Budget)

彈性不是無限資源,應該有明確的上限。

**複雜度預算的三個量化指標:**

| 指標 | 說明 | 建議上限 |
|---|---|---|
| Active flag count | 目前生效中的 feature flag 數量 | 依團隊規模設定,建議 < 2 × 工程師人數 |
| Configuration depth | Config 層次的巢狀深度 | ≤ 3 層 |
| Active API versions | 同時維護的 API 版本數 | ≤ 2 個主版本 |

超出預算時不是「警告」,而是**阻斷**:不允許新增 flag / config 選項,直到舊的被清理。

在 CI pipeline 中用 architecture fitness functions 自動量測:

```yaml
# 範例:在 CI 中檢查 feature flag 數量
check_complexity_budget:
  script:
    - flag_count=$(grep -r "feature_flag" config/ | wc -l)
    - if [ $flag_count -gt $MAX_FLAGS ]; then
        echo "FAIL: Feature flag count $flag_count exceeds budget $MAX_FLAGS";
        exit 1;
      fi
```

這把「彈性擴張」從被動感知(事後才發現太複雜)變成主動阻斷(CI 直接擋住)。

### G.3.4 層次四:可觀測性驅動治理(Observability-First Governance)

**無法觀測的彈性等於無法治理。**

治理需要的可觀測性不是一般的 logging 和 metrics,而是**決策稽核軌跡(Decision Audit Trail)**:

```
每一個「彈性被行使」的事件必須記錄:
    ├── Who:  誰做了這個決策?(人類 or 系統)
    ├── What: 改了什麼?(before / after diff)
    ├── Why:  依據哪個政策版本?有沒有 ticket / ADR?
    ├── When: 什麼時間生效?
    └── Where: 影響哪些環境 / 服務 / 用戶?
```

這不只是 audit log,它是治理的 source of truth。沒有這層,後續的複雜度預算、分層審批都無從建立。

實作模式:把 config 變更視同程式碼部署,走相同的 GitOps 流程:

```
config 變更 PR
    → 自動 diff 展示(影響面分析)
    → reviewer 審核
    → merge → 自動部署到 staging
    → smoke test 通過 → 部署到 prod
    → 變更記錄寫入 audit log
```

> 📌 [圖示建議 G-2]
> *畫一個「決策稽核軌跡」的時間線圖。X 軸是時間,Y 軸是系統配置狀態。每個配置變更點標一個菱形,附上 Who/What/Why。一旁對比一個「無稽核軌跡」的時間線,同樣的時間段只有一條平線,沒有任何可追蹤資訊。*

### G.3.5 層次五:分層審批(Tiered Approval Gateway)

不是所有彈性都需要相同的治理成本。過度治理低風險彈性會讓工程師繞過治理系統;治理缺失高風險彈性是事故前兆。

**風險 × 審批成本矩陣:**

| 風險等級 | 彈性類型 | 治理機制 | 典型例子 |
|---|---|---|---|
| **低** | 無行為影響的配置調整 | 自動批准 + 日誌 | 改 log level、調整 timeout 值 |
| **中** | 改變系統行為路徑 | Peer review + diff 展示 | 開啟 / 關閉 feature flag、角色權限調整 |
| **高** | 影響合規 / 安全 / 架構 | ADR + 人工審核 + staging 驗證 | 資料留存政策、加密方式、外部整合端點 |
| **緊急** | 線上緊急修復 | 事後補文件(< 24h)+ 事後審查 | 生產事故中的緊急 config rollback |

**緊急通道是治理的壓力閥**。完全不允許緊急通道,工程師遇到事故會繞過所有治理;允許緊急通道但要求事後補文件,治理機制才能在壓力下存活。

## G.4 踩坑清單

- **治理工具比被治理的系統更複雜** — 這才是真正的問題。OPA、Cedar、自建 policy engine:如果沒有人維護得了它,它就會成為另一個禁忌神器。先從版本控制下的 YAML 政策檔開始,不要一開始就上最複雜的 policy language。

- **黃金路徑過時了比沒有更糟** — 黃金路徑覆蓋的場景如果已經不代表主流使用模式,工程師每次都要「偏離」,治理成本反而上升。黃金路徑需要每季重新評估。

- **複雜度預算設得太鬆等於沒設** — 如果上限是 500 個 flag 而現在只有 50 個,這個預算沒有任何約束力。預算應設在「讓人感到有點緊張」的位置,才能驅動清理行為。

- **稽核日誌沒有人看** — 光記錄不分析等於白費。稽核軌跡需要有人定期回顧(例如每週的 config review),否則它只是一堆沒人看的 log。

- **緊急通道被濫用** — 如果緊急通道走了 30% 的 config 變更,代表正常流程太慢。解法不是取消緊急通道,而是加速正常流程(自動化 review、縮短 staging 驗證時間)。

- **把「不彈性」當成解法** — 有時最好的治理決策是「把這個配置選項拿掉,直接 hardcode」。彈性有成本,如果沒有真實的業務需求驅動,不要保留彈性點。

## G.5 交付清單

- **彈性點目錄(Flexibility Inventory)**:列出所有 feature flag、config 選項、可配置參數;標注每個的風險等級、最後修改時間、負責人。
- **機制 / 政策邊界圖**:一張圖標示哪些部分是機制層(不能改),哪些是政策層(可改但有 SLA)。
- **黃金路徑文件**:描述標準使用場景的預設配置,標注偏離條件與代價。
- **複雜度預算設定**:三個指標的上限值,以及 CI 中的自動量測 script。
- **決策稽核軌跡格式**:Who / What / Why / When / Where 的記錄 schema。
